Executive Summary

FAST (AI-Assisted Forms Testing) is a CGI-developed internal platform designed to automate the quality assurance process for PDF-based forms and regulated documents. By combining state-of-the-art large language model (LLM) analysis with computer-vision visual diffing and optical character recognition (OCR), FAST dramatically reduces the time and cost associated with manual form review cycles in high-compliance industries such as government services, financial services, and healthcare.

📄

The Problem

Form validation in regulated environments is manual, time-intensive, and error-prone. A single PDF form revision can require hours of side-by-side comparison and stakeholder sign-off.

🤖

The Solution

FAST automates this validation through AI-driven analysis—identifying mismatches, layout anomalies, missing content, and compliance issues with structured, auditable output.

📈

The Value

Reduction in manual QA effort by up to 80%, faster release confidence, and a full traceable audit trail of every test run and finding—ready for compliance review.

Target Personas Government agencies, financial institutions, insurance companies, and any organization that maintains high-volume regulated PDF form libraries and requires rapid, auditable validation on each release cycle.

Features & Capabilities

3.1  Validation Modes

FAST supports three distinct test modes, each addressing a different QA scenario:

🔍

Basic Mode

General-purpose AI validation. The LLM performs an autonomous review of the entire form, checking for spelling errors, formatting anomalies, missing required fields, layout inconsistencies, and compliance-readiness — no custom configuration required.

📋

Specific Mode

User-defined rule-based validation. QA analysts write plain-English validation rules (e.g., "Verify field X matches policy Y", "Confirm all required signatures are present") and FAST applies them precisely via structured prompting.

⚖️

Benchmark Mode

Golden-copy comparison. A reference (approved) PDF is compared against a test form using both AI text analysis and visual diffing. Ideal for regression testing after form revisions. Produces page-by-page similarity scores and change evidence.

LLM Output Structure All three modes produce structured JSON output from the AI model, categorized into: spelling_errors, format_issues, value_mismatches, missing_content, extra_content, layout_anomalies, visual_mismatches, and compliance_issues — each with page number, severity, field name, and evidence text.

3.2  Visual Difference Engine

The visual diff engine converts both PDFs to high-resolution images (200 DPI), performs pixel-level comparison using ImageChops.difference(), and generates a three-panel composite snapshot (expected | actual | diff overlay) for every changed page.

Finding TypeThresholdClassificationColour in Report
No material change< 0.10% pixel diffMatchOK
Minor visual difference0.10% – 1.99%WarningWARN
Significant content change≥ 2.0%MajorBAD
Signature field changeSpatial proximity + keyword matchSignature CandidateREVIEW

Signature Detection Algorithm

  • Keyword scan per page: PRESIDENT, SECRETARY, SIGNATURE
  • Spatial zone validation: change must appear in the lower 45% of the page
  • Bounding box proximity: within 120px horizontal and 70px vertical of keyword anchor
  • Compactness check: diff area must be < 3.0% of page (eliminates false positives)
  • Confidence levels: high (PRESIDENT/SECRETARY, ≤ 1.0% diff), medium (generic SIGNATURE)

3.3  OCR & Intelligent Text Extraction

Digital PDFs

  • Embedded text extraction via pdfplumber
  • Per-page text segmentation for granular analysis
  • AcroForm field enumeration (field name + value)
  • Automatic detection of scanned vs. digital content

Scanned / Legacy PDFs

  • OCR fallback via Tesseract at 300 DPI for scanned pages
  • pdf2image Poppler conversion pipeline
  • Per-page OCR with configurable DPI
  • Extraction metadata included in LLM prompt context

3.4  Reporting & Audit Trail

After each test run, FAST generates a standalone HTML report that can be shared with stakeholders, embedded in portals, or archived for compliance purposes. Reports are self-contained (no external dependencies) and include embedded visual evidence.

📊

Decision Matrix

Per-page table with columns: Status, Severity, Decision, Evidence, Similarity Score, and visual Snapshot link. Filterable by Mismatch / Review / Match.

🎯

Severity Classification

Four-level severity system: Critical → High → Medium → Low, derived from the AI finding type and page-level visual analysis. Each page receives a deterministic verdict.

🗂️

Run History

Full execution history with pass/warn/error counters per run. Individual result drill-down per test case. All reports are persisted on disk and remain accessible after run completion.

📦

Evidence Bundling

Three-panel PNG snapshots embedded in reports. Direct links to source PDFs for side-by-side review. Summary narrative generated by the AI to accompany findings.

3.5  Project & Form Management

  • Multi-project workspace: each project encapsulates its own forms, test cases, and run history
  • PDF upload with automatic duplicate-name resolution and strict .pdf extension enforcement
  • Form version tracking via naming convention (filename-based)
  • Benchmark (golden copy) form designation per test case
  • Test case library: create, view, and delete test cases independently of run execution
  • Bulk test execution: select multiple test cases and execute them in a single run
  • Real-time run status polling endpoint for UI progress tracking
  • Run dashboard: per-project summary of latest run results at a glance

4.2  Test Execution Data Flow

1
User triggers a Run

Analyst selects one or more test cases and clicks Execute. A Run record is created in the database with status pending. The web server returns immediately; processing executes synchronously per test case.

2
PDF Text Extraction

The extractor.py module reads the form PDF, extracts embedded text via pdfplumber, and enumerates AcroForm fields. If the document is detected as scanned, Tesseract OCR is invoked at 300 DPI as a fallback.

3
Visual Diff Generation (Benchmark mode)

For benchmark test cases, visual_diff.py converts both PDFs to images and performs pixel-level comparison. Three-panel snapshots are saved to instance/visual_diffs/. Per-page similarity scores and signature candidates are computed.

4
Prompt Construction

prompt_builder.py assembles a structured prompt containing the extracted text, per-page text array, form fields, visual diff evidence, and mode-specific validation instructions. A deterministic system message ensures consistent JSON output.

5
LLM Analysis

The prompt is sent to the Gemini API (temperature = 0.0 for maximum determinism). The model returns a structured JSON object containing categorized findings, severity levels, page references, and an overall summary narrative.

6
Reconciliation & Persistence

The runner reconciles visual findings with LLM output, recomputes summary counts, derives pass/warn/error metrics, and persists the full result_json to the database alongside the generated HTML report path.

7
HTML Report Generation

html_report.py generates a standalone HTML document with the decision matrix, severity chips, visual snapshot links, and summary metrics. The report is written to instance/reports/ and served via a dedicated endpoint.

Current Limitations
MVP Scope Notice FAST is currently in an MVP (Minimum Viable Product) state. The limitations below are known, documented, and addressed in the enterprise roadmap. They do not reflect the target production architecture.

5.1  Capacity & Scale Constraints

Form Upload Limit — 10 Forms per Project

Hard-coded to 10 PDF forms per project. Exceeding this limit blocks further uploads. This is a development convenience guard, not a platform constraint — trivially configurable or removed in production.

File Size — No Enforcement

There is no maximum PDF file size validation on upload. Large PDFs (100+ pages, embedded images) will consume significant memory during OCR and visual diff generation, potentially causing timeouts or process crashes.

Serial Test Execution — No Concurrency

Test cases within a run are processed serially. There is no parallel execution engine or job queue. For large runs (>5 complex test cases), execution time can extend into several minutes.

Results History — 20-Run Display Limit

The results history view is limited to displaying the 20 most recent runs per project. Older records persist in the database but are not accessible via the UI without modification.

5.2  Security & Authentication

Stub Authentication — No Password Validation

The current login mechanism accepts any username/password combination. There is no real identity provider integration, password hashing, or user database. Not suitable for production without replacement.

No Role-Based Access Control (RBAC)

All authenticated sessions have full access to all projects and data. There are no user roles (admin, analyst, read-only) and no project-level access restrictions.

Hardcoded Flask Secret Key

The Flask SECRET_KEY is set to the literal string "dev" in the application factory. Session cookies are cryptographically weak in this configuration.

No CSRF Protection

Form submissions do not include CSRF tokens. In a multi-user production deployment, this is a cross-site request forgery vulnerability.

5.3  Infrastructure & Persistence

SQLite — Not Production-Scale

SQLite provides no network access, no concurrent write support, and limited scalability. In a containerized or multi-instance deployment, a proper RDBMS (PostgreSQL, Amazon RDS) is required.

Local File System Dependency

Uploaded PDFs, generated reports, and visual diff images are stored on the local file system in the instance/ directory. This is incompatible with horizontal scaling or container restarts without a persistent volume or object storage (e.g., Amazon S3).

No LLM Request Timeout or Retry Logic

There is no timeout configured for Gemini API calls, and no retry logic for transient failures. A slow or failed API response will block the entire run execution thread.

5.4  Functional Gaps

No REST API

FAST is currently a web-UI-only product. There are no REST or GraphQL endpoints for programmatic integration with CI/CD pipelines, test automation frameworks, or third-party tools.

No Scheduling or Automation

Test runs must be triggered manually via the UI. There is no scheduling mechanism, webhook trigger, or CI/CD integration hook.

No Audit Logging

User actions (login, upload, delete, run trigger) are not logged to an audit trail. This is a gap for compliance-regulated environments that require immutable access logs.

Single LLM Provider Dependency

The platform is tightly coupled to the Google Gemini API. A provider outage or deprecation directly impacts availability. The AWS Bedrock migration path (Section 6) resolves this.

6.2  AWS Components & Services

🧠

Amazon Bedrock

Replaces Gemini as the LLM provider. Supports Claude 3.5/3.7 Sonnet, Amazon Nova, and Titan models via a unified API. Provides enterprise SLA, data residency, and no data egress to third-party providers. Enables model switching without code changes.

🐳

Amazon ECS Fargate

Serverless container runtime for the FAST web application and background worker containers. Auto-scales task count based on SQS queue depth. No EC2 instance management required. Integrates with ALB for load balancing and ACM for TLS termination.

🗄️

Amazon RDS (PostgreSQL)

Replaces SQLite with a fully managed relational database. Multi-AZ deployment for high availability. Supports connection pooling via RDS Proxy. SQLAlchemy ORM requires only a connection string change.

🪣

Amazon S3

Object storage for uploaded PDFs, generated HTML reports, visual diff PNGs, and OCR artifacts. Replaces the local instance/ directory. Enables pre-signed URL generation for secure, time-limited report sharing.

🔐

Amazon Cognito

Managed identity provider replacing the stub auth system. Supports SAML 2.0 federation with enterprise IdPs (Active Directory, Okta), OAuth 2.0 / OIDC flows, MFA enforcement, and role-based group assignments.

📨

Amazon SQS + Workers

Decouples test run submission from execution. The web app enqueues jobs; ECS Fargate worker tasks consume and process them asynchronously. Enables parallel execution and prevents UI timeouts on large runs.

🔑

AWS Secrets Manager

Securely stores and rotates API keys, database credentials, and Flask secret keys. Containers retrieve secrets at runtime via IAM role — no secrets in environment variables or container images.

📊

Amazon CloudWatch

Centralized logging, metric collection, and alerting. Container logs (Flask, worker) stream directly. Custom metrics for run duration, LLM latency, error rates. Dashboards for operational visibility and SLA reporting.

6.5  Bedrock LLM Client Integration

Replacing llm_client.py's Gemini call with Amazon Bedrock requires only a lightweight adapter swap. The prompt schema, JSON output contract, and system message remain identical:

import boto3, json

class BedrockLLMClient:
    def __init__(self):
        self.client = boto3.client("bedrock-runtime", region_name="us-east-1")
        # Drop-in replacement for Gemini — same model_id env var pattern
        self.model_id = os.getenv("BEDROCK_MODEL_ID",
                                  "anthropic.claude-3-5-sonnet-20241022-v2:0")

    def validate(self, system_prompt: str, user_prompt: str) -> dict:
        body = json.dumps({
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 8192,
            "temperature": 0.0,
            "system": system_prompt,
            "messages": [{"role": "user",
                          "content": user_prompt}]
        })
        response = self.client.invoke_model(
            modelId=self.model_id,
            body=body,
            contentType="application/json",
            accept="application/json"
        )
        result = json.loads(response["body"].read())
        return json.loads(result["content"][0]["text"])

6.6  Infrastructure as Code (AWS CDK — Python)

# cdk_stack.py — FAST on AWS ECS + Bedrock (abbreviated)
from aws_cdk import (
    Stack, aws_ecs as ecs, aws_ecs_patterns as patterns,
    aws_rds as rds, aws_s3 as s3,
    aws_sqs as sqs, aws_cognito as cognito
)

class FastStack(Stack):
    def __init__(self, scope, id, **kwargs):
        super().__init__(scope, id, **kwargs)

        # S3 — persistent storage for uploads, reports, diffs
        bucket = s3.Bucket(self, "FastAssets",
            versioned=True,
            encryption=s3.BucketEncryption.S3_MANAGED)

        # RDS PostgreSQL — replaces SQLite
        db = rds.DatabaseInstance(self, "FastDB",
            engine=rds.DatabaseInstanceEngine.postgres(
                version=rds.PostgresEngineVersion.VER_16),
            multi_az=True,
            deletion_protection=True)

        # SQS — job queue for async test execution
        queue = sqs.Queue(self, "FastJobQueue",
            visibility_timeout=Duration.minutes(15))

        # Cognito — replace stub auth
        pool = cognito.UserPool(self, "FastUsers",
            mfa=cognito.Mfa.REQUIRED,
            password_policy=cognito.PasswordPolicy(min_length=12))

        # ECS Fargate — web service with ALB
        web_service = patterns.ApplicationLoadBalancedFargateService(
            self, "FastWeb",
            task_image_options=patterns.ApplicationLoadBalancedTaskImageOptions(
                image=ecs.ContainerImage.from_asset(".",
                    file="Dockerfile.web")),
            cpu=1024, memory_limit_mib=2048,
            desired_count=2)

        # Grant Bedrock inference permissions
        web_service.task_definition.task_role.add_managed_policy(
            iam.ManagedPolicy.from_aws_managed_policy_name(
                "AmazonBedrockFullAccess"))
